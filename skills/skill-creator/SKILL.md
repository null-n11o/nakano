# Skill Creator

Claude のスキルを設計・作成・改善するための対話型ガイド。

---

## スキルとは

スキルは、Claudeに特定のタスクやワークフローを教えるフォルダ構成のパッケージ。一度定義すれば、Claude.ai / Claude Code / API のすべてで同じように動作する。

### フォルダ構成

```
your-skill-name/         # kebab-case で命名
├── SKILL.md             # 必須 - メイン定義ファイル
├── scripts/             # オプション - Python, Bash 等の実行スクリプト
├── references/          # オプション - 参照ドキュメント
└── assets/              # オプション - テンプレート, アイコン等
```

### プログレッシブディスクロージャー（3段階読み込み）

1. **YAML frontmatter** — 常にコンテキストに存在。Claudeがスキルを使うか判断する
2. **SKILL.md 本文** — スキルが関連すると判断したとき読み込む
3. **references/ 等のリンクファイル** — 必要に応じて読み込む

## 作成プロセス

### Step 1: ユースケースを定義する

まずユーザーに以下を確認する（一度に全部聞かず、最重要から順番に）：

1. このスキルで何を実現したいか？
2. どんな言葉を言ったときに使われるべきか？（トリガーフレーズ）
3. 期待する出力形式は？
4. 既存のワークフローをスキル化する場合、会話履歴から手順を抽出する

**良いユースケース定義の例：**
```
ユースケース: スプリントプランニング
トリガー: "スプリントを計画して" "タスクを作成して"
手順:
  1. Linear から現在のプロジェクト状況を取得
  2. チーム速度を分析
  3. タスク優先度を提案
  4. Linear にタスクを作成
```

### Step 2: SKILL.md を書く

#### YAML frontmatter（最重要）

```yaml
---
name: your-skill-name
description: スキルの説明。[何をするか] + [いつ使うか（トリガー条件）]。
---
```

**name のルール：**
- kebab-case のみ（スペース・大文字・アンダースコア不可）
- フォルダ名と一致させる
- `claude` / `anthropic` を含めない（予約語）

**description のルール：**
- **必ず「何をするか」と「いつ使うか」の両方を含める**
- 1024文字以内
- XML タグ（`<` `>`）不可
- ユーザーが実際に使う言葉（トリガーフレーズ）を含める

**良い description の例：**
```yaml
description: Figma デザインファイルを解析し、開発者向けハンドオフドキュメントを生成する。
  .fig ファイルをアップロードしたとき、「デザインスペック」「コンポーネントドキュメント」
  「デザイン→コード変換」と言ったときに使用する。
```

**悪い description の例：**
```yaml
# NG: 曖昧すぎる
description: プロジェクトの作業を支援する。

# NG: トリガーがない
description: 複雑なドキュメントシステムを作成する。
```

#### SKILL.md 本文の推奨構成

```markdown
# スキル名

## 概要
[スキルの目的を1-2文で]

## 手順
### Step 1: [最初のステップ]
[具体的な説明]

### Step 2: [次のステップ]
[具体的な説明]

## 使用例
### 例1: [よくあるシナリオ]
ユーザーが言う: "..."
処理:
  1. ...
  2. ...
結果: ...

## トラブルシューティング
### エラー: [よくあるエラー]
原因: ...
解決策: ...
```

**本文を書くときのポイント：**
- 具体的・実行可能な指示を書く（曖昧な表現を避ける）
- 重要な手順は箇条書き・番号付きリストを使う
- エラーハンドリングを含める
- 詳細な参照資料は `references/` に移動し、SKILL.md はコンパクトに保つ

## 成功基準

スキルが機能しているか確認する指標：

**定量指標：**
- 関連クエリの 90% でスキルが自動トリガーされる
- 同じタスクで比較してスキルなしよりトークン消費が少ない
- API エラーが 0

**定性指標：**
- ユーザーが次のステップを自分で指示する必要がない
- 複数回実行して一貫した結果が出る
- 初回ユーザーでも最小限のガイダンスでタスクを完了できる

## テスト方法

### トリガーテスト

スキルが適切なタイミングで読み込まれるか確認：

```
✅ トリガーすべきクエリ:
  - "新しい[サービス名]ワークスペースを設定して"
  - "[サービス名]でプロジェクトを作りたい"
  - "Q4 計画の[サービス名]プロジェクトを初期化して"

❌ トリガーすべきでないクエリ:
  - "東京の天気は？"
  - "Python コードを書いて"
```

デバッグ方法：Claude に「[スキル名] スキルはどんなときに使いますか？」と聞くと、description の内容を引用して答える。

### 改善サイクル

スキルは生きたドキュメント。以下のシグナルに基づいて改善を続ける：

| シグナル | 原因 | 対処 |
|---------|------|------|
| スキルが自動起動しない | description が曖昧 | トリガーフレーズを追加・具体化 |
| 関係ないときにも起動する | description が広すぎる | ネガティブトリガーを追加 |
| 手順通りに動かない | 指示が不明確 | 具体的な例・コマンドを追加 |
| コンテキストが重い | SKILL.md が大きすぎる | 詳細を `references/` に移動 |
